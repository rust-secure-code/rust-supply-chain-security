About this guide

This guide is aimed at both developers working on Rust programs for end-users and libraries developers.

It focuses on currently available tooling for the Rust ecosystem, but also mentions more general advice or tooling. It has a hands-on focus and tries to provide actionnable items whenever possible. It will also mention the current limitations and missing parts. It builds upon existing resources, like llogiq's blog post.

Using dependencies

cargo makes it really easy to use dependencies, so the Rust ecosystem tends to use a lot of them. This comes with security risks, and we'll see how we can mitigate them.

Choosing dependencies

Risks

  • vulnerabilities
  • malicious crates or releases
    • typo squatting (e.g. rustdecimal)
    • build.rs/proc macros risks
  • (lack of) maintenance
  • license compliance (not directly linked to security but shared tooling)

Minimizing dependencies

  • minimize dependencies (limit features, tools to explore dependency tree, etc.)
  • number of dependencies != attack surface (cargo supply-chain)

Assessing a dependency

  • evaluate a crate
    • check list by Embark
  • cargo crev
  • cargo vet
  • unsafe code management (cargo-geiger, etc.)

Maintaining dependencies

  • Cargo.lock for binaries
  • vuln monitoring with cargo audit/deny
  • regular updates (to stay on maintained versions)
    • cargo outdated
    • dependabot/github security alerts https://github.blog/2022-06-06-github-brings-supply-chain-security-features-to-the-rust-community/

Software Bill Of Materials

SBOM for Rust programs usually need to be created from the sources.

Choosing a SBOM format is up to you but they should be mostly interoperable.

CycloneDX

There is an official CycloneDX cargo subcommand to generate a SBOM from a Rust project source, cargo-cyclonedx

cargo cyclonedx --all --format json --manifest-path Cargo.toml

It can handle workspaces, and produces a json or XML file.

To aggregate several files, validate or sign them, you can use the cyclonedx-cli.

cyclonedx merge --input-files *.json --output-file myprogram.cdx.json

  • cargo auditable
  • rust version in binary
  • trivy, other tooling...
  • warning not everything is visible: embedded c/c++ deps not visible, etc.

Running in production

In this section, we will see what we can do to track known vulnerabilities in Rust programs.

It won't cover generic deployment security as there are numerous options (containers, Windows msi/exe installers, Linux rpm/dpkg/etc. packages, etc.) but you should make sure you can verify that deployed artifacts' source and integrity (various artifact signature mechanism can help).

Audit binaries

cargo audit has some capacities to audit Rust binaries for known vulnerabilities in their dependencies.

It has two operation modes in this context:

  • Audit for known vulnerabilities in the dependency list embedded by cargo-auditable.
  • Audit for known vulnerabilities in a dependency list extracted from panic messages.
    • This is incomplete as not all crates include panics
    • Depending on the compilation options, panic messages may not be available

You can also use other tools like trivy which is able to look for known vulnerabilities in Rust binaries by using the same dependency list built by cargo-auditable.
